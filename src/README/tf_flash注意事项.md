# TF Flash 高精度定位融合记录及注意事项

## 1. `/tf_flash` 话题数据含义与物理直觉

`/tf_flash` 话题通常使用 `std_msgs::String` 格式发布，内容格式为例：
`dist=%f dx=%lf dy=%lf yaw(rad)=%lf yaw(deg)=%lf`

从物理直觉上讲，这是**以机器人当前需要停靠的“目标靶点”为原点建立的局部坐标系**（也可以认为是对接时的 World Frame）。其各项数据代表机器人“离完美对接状态”分别在各个自由度上的空间偏差。

* **`dist` (距离绝对方位差)**: 机器人当前状态相对于完美停靠点（零点）的**绝对直线距离模长**。
  * **物理直觉**：你可以把它想象成一根栓在机器人中心和终点之间的松紧带长度，无论左右前后偏差多少，它综合反映了“我们离目标点在空间上绝对隔了多远”。常用于快速条件判定。
* **`dx` (X轴空间偏差)**: 机器人在对接坐标系 X 轴方向上的偏差（单位：米）。
  * **物理直觉**：这是车辆相较于对接靶点的“前后”位移误。如果小车正在往前靠靠站，未贴合时就是距离差，开过头了正负号会随之翻转。
* **`dy` (Y轴空间偏差)**: 机器人在对接坐标系 Y 轴方向上的偏差（单位：米）。
  * **物理直觉**：这代表小车相较于靶点的“左右偏离”误差。直观体现为你的小车“没有正对目标线，而是偏左或者偏右了多少米”。
* **`yaw(rad)` (偏航角误差，弧度制)**: 机器人车体朝向，与目标完美对接朝向之间的角度误差。
  * **物理直觉**：代表“你的车头歪没歪”。哪怕小车的中心坐标正好停在了目标点（dx=0, dy=0），如果车是斜着停的，`yaw` 的值就不为 0。在原生逻辑里，`yaw > 0` 往往意味着车头顺时针歪了。如果是数学系应用，可能要乘上 -1 转为标准的逆时针。
* **`yaw(deg)` (偏航脚误差，角度制)**: 与 `yaw(rad)` 一样，但这纯粹是为了以人的直觉看懂（如：车头歪了5°），给算法监控和 Debug 日志用的。

## 2. 融入其它工程以实现 5mm 级精准对接的步骤与核心方案

复刻这种通过传感器提供的高精度偏差，引入至新项目中并实现毫米级收敛泊车，必须采用具备**带微分阻尼**的系统去接管纯几何的跟踪。详细步骤如下：

### 第 1 步：速度解算与一阶低通滤波（至关重要）
视觉或激光定位提供出的距离偏差（哪怕到了 5mm 级），都会携带微小高频噪声。越靠近终点抖动越明显。 
需要根据连续两帧的位置差算出相对速度 `V` 并去抖：
```cpp
V_instant = (pos_now - pos_prev) / delta_time;
V_filtered = alpha * V_instant + (1.0 - alpha) * V_filtered; // 低通滤波
```

### 第 2 步：外部误差坐标系映射到车体（World -> Body）
无论目标点定义的 XYZ 怎么偏，作为执行器的小车必须要知道自己在它**自身的**前向和侧向还有多大误差。需要利用 `odom` 里面本身的车头偏航角进行转换：
```cpp
double c = std::cos(odom_yaw), s = std::sin(odom_yaw);
double ex_b =  c * target_dx + s * target_dy;  // 小车需要向前补的位移
double ey_b = -s * target_dx + c * target_dy;  // 小车需要侧向微调的位移
```

### 第 3 步：加入 PID 阻尼控制 (“软着陆”)
单纯的位置弥补会导致震荡，越界了再拉回来，反反复复无法靠站停稳。此时，必须要加入底层的微分阻尼做刹车力：
```cpp
// 终点控制输出 = 基础移动前馈 + 位置修正向位移牵引 - 阻尼刹车(与车速方向相反)
target_vx = V_forward + (Kp_pos * ex_b) - (Kd_damp * odom_vx); 
```

### 第 4 步：制定极严苛的三维同步停车条件
不能单纯 `dist < 某值` 就直接把车关了休眠，这在惯性下往往会继续滑动脱靶。高精度闭环必须同时满足三重死区：
1. **距离到点**：`dist < 0.007` 米 (即误差进入半径 7mm 内)。
2. **车头摆直**：`abs(yaw_rad) < 0.02` 弧度 (约 1.1 度)。
3. **车辆静止**：`速度模长 < 0.01` m/s。 
只有三者同时在收敛区间，才真正下发结束命令。

## 3. 防坑项：超时安全接管（Timeout）
高精度的测距设备存在视野遮挡或者解算失败导致中断的情况。**切忌将 `TF` 信号作为一个无限期信任的前馈源持续控制小车**。
必须设置安全接管（例如超时判断时间设为 0.2 秒）：
```cpp
// 当距离上一次获得定位刷新超过 0.2秒时：
if ((ros::Time::now() - tf_flash_stamp_).toSec() > 0.2) {
    // 强制执行降速、指数衰减或者原点抱死刹车
    vx = 0; vy = 0;
}
```
以免在最后关头瞎往前冲导致测试设备撞毁。

## 4. 补充注意事项与调参经验

### 4.1. 远近自适应限幅与分段前馈
在真实的融合项目中，不可始终保持固定的前馈速度和最大速度限幅（如果前向 P 加上常数前馈导致速度太大，近距离下会直接冲撞靶点）。
**项目实现建议遵循代码中的分段机制：**
* **远距离 (> 10cm)**：可以开启较大的最高限幅，比如 `v_max = 0.5 m/s`。
* **近距离 (< 10cm)**：将整体输出紧紧掐缩在低速范围内，如 `v_max = 0.15 m/s`。
```cpp
// 强烈限制微调区的最大运动速度
double err_norm = std::hypot(tf_dx_, tf_dy_);
double vmax = (err_norm < 0.1) ? 0.15 : vmax_global;
clampNormAndScale(ux, uy, 0.0, vmax);
```

### 4.2. 反馈源的选择差异：为何差分速度要转系？
大部分人在做 PD 闭环时，往往习惯直接拿轮式里程计 `odom_vx / odom_vy` 来当作速度 D 阻尼反馈。
然而，在毫米级对接最后阶段，底盘轮胎极可能发生微小打滑或者摩擦扭曲！这时候 `odom` 认为机器已经停了，不再有速度反馈，但相机视角的机器可能还在滑动（也就是微小过冲阶段）。
* **高端做法**：代码中用的是 `tf_vx_raw_` 和 `tf_vy_raw_`（TF求得并低通滤波后的World系统速度），并将其通过 $\sin/\cos(yaw)$ 转换到了车体本身的 `vx_fb` 和 `vy_fb` 来担任微分阻尼项。这相当于利用**定位系统反馈回来的真实物理移动速度**来阻尼掉惯性，而非纯盲目的底盘速度。

### 4.3. 时间同步（Time Sync）问题
将一个独立的高精度传感网络（可能是另一台电脑处理出的视觉坐标）并入本系统时，确保坐标解析的时间戳：
* 如果外部数据卡顿，上述提到的 `dt` 计算速度会出现极大偏差。
* 计算 `V_instant = (pos_now - pos_prev) / dt` 时，必须确保此 `dt` **不得等于0** 并且**在合理区间内**（如项目中判断的 `dt > 0.005 && dt < 1.0` 才计算差分）。如果是卡死的跳帧老信号或者隔了很久突然来的骤变信号，此帧速度不能拿来反馈或低通。
