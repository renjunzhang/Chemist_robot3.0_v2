# 当前项目旧方案总结

## 编译注意事项

1. 当前仓库根目录直接用默认 profile 执行 `catkin build` 会先卡在工作区路径不一致。
2. 原因是旧 `build/.catkin_tools.yaml` 里记录的工作区还是 `/home/a/chemist_robot3.0`，而当前实际路径是 `/data/a/chemist_robot3.0`。
3. 如果不想清理原有 `build/`、`devel/`、`logs/`，建议新建独立 profile 再编。
4. 当前已经验证可用的非侵入式编译方式是：
   - profile：`data_build`
   - build space：`build_data/`
   - devel space：`devel_data/`
   - log space：`logs_data/`
5. 当前 `data_build` profile 里临时 skip 了 `web_video_server`，原因是它缺少 `async_web_server_cpp` 依赖。
6. `web_video_server` 主要是 ROS 图像话题的 HTTP/网页视频流服务，不是底盘、定位、导航主链的必需包。
7. 在跳过 `web_video_server` 后，当前项目其余 48 个参与构建的包已经成功通过 `catkin build`。
8. 如果后面需要使用这套独立编译结果，记得重新 source：
   - `/data/a/chemist_robot3.0/devel_data/setup.bash`

## 从 0 部署依赖

下面这部分只记录当前仓库从 0 编译时，外部环境至少需要具备哪些依赖。

### 1. 基础环境

建议基线：

1. Ubuntu 20.04
2. ROS Noetic
3. `catkin_tools`
4. `rosdep`

建议先准备：

```bash
sudo apt update
sudo apt install -y ros-noetic-desktop-full python3-catkin-tools python3-rosdep
```

如果本机还没初始化 `rosdep`，还需要：

```bash
sudo rosdep init
rosdep update
```

### 2. 已验证需要的系统 / 开发库

这些依赖来自当前工作区 `package.xml`、`CMakeLists.txt` 和实际编译结果，当前环境已经验证可用：

```bash
sudo apt install -y \
  python3-dev python3-numpy python3-opencv python3-pykdl \
  libeigen3-dev libboost-all-dev libyaml-cpp-dev \
  liblapack-dev libblas-dev libsuitesparse-dev libbullet-dev \
  libopencv-dev libpcl-dev libvtk7-qt-dev \
  libsdl1.2-dev libsdl-image1.2-dev libnetpbm10-dev \
  ffmpeg libavcodec-dev libavformat-dev libavutil-dev libswscale-dev
```

### 3. 传感器 / 相机相关额外依赖

当前仓库里的 `locator` 还依赖：

1. `realsense2_camera`
2. `realsense2` SDK / 开发库
3. OpenGL

这部分在仓库里能直接看到：

1. [locator/package.xml](/data/a/chemist_robot3.0/src/locator/package.xml)
2. [locator/CMakeLists.txt](/data/a/chemist_robot3.0/src/locator/CMakeLists.txt)

部署时建议确认本机已经具备：

```bash
sudo apt install -y libgl1-mesa-dev libglu1-mesa-dev freeglut3-dev
```

`realsense2_camera` 和 `librealsense2` 需要按你现场使用的 RealSense 版本额外安装。

### 4. Python 运行时依赖

这部分主要来自 `robot_message_bridge`、GUI、MQTT、Flask 和语音播报脚本，不一定都是“纯编译必需”，但从 0 部署时通常会一起用到。

仓库脚本里明确能看到的第三方 Python 依赖包括：

1. `requests`
2. `flask`
3. `flask_executor`
4. `gevent`
5. `paho-mqtt`
6. `pyserial`
7. `pyttsx3`
8. `func_timeout`
9. `PyQt5`
10. `PyQtWebEngine`

建议至少准备：

```bash
sudo apt install -y \
  python3-requests python3-flask python3-gevent \
  python3-paho-mqtt python3-serial python3-pyqt5 \
  python3-pyqt5.qtwebengine
```

下面这些在不同机器上更常见的安装方式是 `pip3`：

```bash
pip3 install flask-executor func-timeout pyttsx3
```

### 5. `web_video_server` 是可选依赖

如果只是跑当前主链，`web_video_server` 可以先跳过。

它额外需要：

1. `async_web_server_cpp`
2. `ffmpeg` 相关开发库

其中 `ffmpeg` 相关库已经在上面的系统依赖里列了；当前真正缺的是 `async_web_server_cpp`。

注意：

1. 当前环境里 `rosdep` 对 `async_web_server_cpp` 没有可直接解析的规则
2. 所以如果后面要启用 `web_video_server`
3. 需要单独安装或把它的源码放进工作区

### 6. 推荐安装顺序

建议从 0 部署按下面顺序走：

1. 先装 Ubuntu 20.04 + ROS Noetic
2. 再装 `catkin_tools` 和 `rosdep`
3. 再装上面列出的系统 / 开发库
4. 再装 Python 运行时依赖
5. 如果需要 RealSense，再单独补 `realsense2_camera` 和 `librealsense2`
6. 如果需要网页视频流，再单独补 `async_web_server_cpp`

## 说明

这份文档只总结当前项目里已经存在的结构、节点关系、接口和运行链路。

不讨论新方案，不讨论替换路线，不给改造建议。

## 1. 项目整体结构

当前项目可以按功能分成 4 层：

1. 上层业务与 DMS 通信层
2. 导航业务执行层
3. 地图 / 定位 / 导航内核层
4. 底盘与传感器驱动层

对应目录主要是：

```text
src/
├── robot_message_bridge/
├── platform/
│   ├── control/
│   ├── hf_mapping/
│   └── hf_nav/
```

## 4. 底盘与传感器驱动层

### 4.1 串口底盘控制

当前底盘控制由 `communication_rs485` 完成。

关键节点行为：

1. 订阅 `twist_mux/cmd_vel`
2. 发布 `platform_info`
3. 把 ROS `Twist` 转成 RS485 协议速度和控制位

当前速度语义：

1. `linear.x`：前后速度
2. `linear.y`：横移速度
3. `angular.z`：旋转速度
4. `angular.x == 1`：充电标志位
5. `angular.y == 1`：清除底盘里程计标志位

### 4.2 `twist_mux`

当前 `twist_mux` 负责多路速度仲裁。

已配置输入：

1. `crg_vel`
2. `nav_vel`
3. `joy_vel`

当前优先级：

1. joystick：200
2. navigation：100
3. recharge：80

### 4.3 安全雷达与激光处理

当前 `hf_bringup.launch` 里包含：

1. `sick_safetyscanners`
2. 机器人本体到前后雷达的静态 TF
3. 虚拟拼接雷达 `scan_merge`
4. `ira_laser_tools` 激光拼接
5. `laser_filters` 激光滤波

当前导航和建图实际使用的激光主题是：

1. `/scan_full_filtered`

## 5. 建图层

当前建图包是 `hf_karto`。

`slam_karto` 的输入和关键参数：

1. `scan` 重映射到 `/scan_full_filtered`
2. `odom_frame = /odom`
3. `base_frame = /hf_base_link`
4. 地图分辨率 `0.02`

当前建图链是独立于 `move_base` 的。

## 6. 定位与导航内核层

### 6.1 定位部分

当前 `hf_navigation.launch` 中的定位链包括：

1. `map_server`
2. `cartographer_node`
3. `lidar_odometry_node`

当前用途：

1. 读取已有地图
2. 基于 `cartographer` 做定位
3. 发布较快的位姿结果 `/amcl_pose_tf`

### 6.2 当前导航内核

当前导航内核就是原生 `move_base`。

启动文件：

1. `platform/hf_nav/hf_navigation/launch/include/move_base.launch`

当前配置：

1. `base_global_planner = global_planner/GlobalPlanner`
2. `base_local_planner = eband_local_planner/EBandPlannerROS`
3. `cmd_vel` 重映射到 `/hf_platform/nav_vel`
4. `odom` 重映射到 `/odom`

### 6.3 当前 costmap 结构

`move_base` 当前加载：

1. `costmap_common_params.yaml`
2. `map_nav_params/local_costmap_params.yaml`
3. `map_nav_params/global_costmap_params.yaml`
4. `move_base_params.yaml`
5. `local_planner/eband_local_planner_params.yaml`

当前 common costmap 关键参数：

1. footprint：`[[0.52, -0.42], [0.52, 0.42], [-0.52, 0.42], [-0.52, -0.42]]`
2. obstacle source：`/scan_full_filtered`
3. obstacle layer：`costmap_2d::ObstacleLayer`
4. inflation layer：`costmap_2d::InflationLayer`
5. inflation radius：`0.9`

### 6.4 当前局部规划器配置特点

当前 local planner 是 `EBandPlannerROS`。

参数特点：

1. `xy_goal_tolerance = 0.005`
2. `yaw_goal_tolerance = 0.005`
3. `differential_drive = false`
4. `rotation_correction_threshold = 20`

## 7. 导航业务执行层

当前导航业务执行层是 `task_communication`。

核心节点：

1. `platform_communication_node`

当前职责：

1. 接收 DMS 导航命令
2. 查询站点表
3. 把目标站点转换成 `MoveBaseGoal`
4. 驱动 `move_base`
5. 回传导航状态
6. 处理充电、解除充电、后退、旋转、充电前对位等流程

### 7.1 当前业务接口

当前 `platform_communication_node` 对外提供：

1. 订阅 `/obsNavigation_in`
2. 发布 `/obsNavigation_out`
3. 提供 `/inquiryNavigation_out`
4. 发布 `/voide_broadcast_msg`
5. 订阅 `/amcl_pose_tf`

### 7.2 当前与 move_base 的关系

当前 `platform_communication_node` 内部是直接连接 `/move_base` action 的。

也就是说：

1. 业务层不是通过抽象导航接口调用导航
2. 而是直接依赖 `move_base_msgs::MoveBaseAction`

### 7.3 当前业务动作

当前业务动作至少包括：

1. `move`
2. `charge`
3. `uncharge`

### 7.4 当前特殊行为

当前旧链里这些特殊行为也都通过 `MoveBaseGoal` 完成：

1. `StrollToGoal`
2. `StrollToBack`
3. `RotateInPlace`
4. `ReCharge`
5. `StrollToCharge`

### 7.5 当前充电保持逻辑

当前 `ReCharging()` 线程会：

1. 读取 `/hf_platform/platform_info`
2. 判断电量和充电状态
3. 通过 `/hf_platform/crg_vel` 持续发充电保持命令

这部分不走 `move_base`，而是直接走速度控制链。

## 8. DMS / 上位机通信层

当前 DMS 相关逻辑主要在 `robot_message_bridge/scripts/dms_to_robot.py`。

### 8.1 当前职责

主要职责：

1. 读取机器人参数配置
2. 建立 MQTT / Flask / DMS 通信
3. 发布导航和操作话题
4. 查询导航和操作结果
5. 发布底盘锁定与手柄速度

### 8.2 当前导航相关 ROS 接口

当前脚本使用的导航相关接口：

1. 发布 `/obsNavigation_in`
2. 等待 `/obsNavigation_out`
3. 调用 `/inquiryNavigation_out`
4. 发布 `/hf_platform/pause_navigation`
5. 发布 `/hf_platform/joy_vel`

### 8.3 当前系统节点检查

脚本中维护了一组当前系统依赖节点清单，例如：

1. 前后雷达
2. 底盘节点
3. 地图节点
4. `platform_communication_node`
5. 机械臂节点

## 9. 当前完整运行链

当前典型运行链可以概括成下面这样：

```text
[DMS / Flask / MQTT]
   |
   v
[robot_message_bridge/dms_to_robot.py]
   |
   v
/obsNavigation_in
   |
   v
[task_communication/platform_communication_node]
   |
   v
/move_base (MoveBaseAction)
   |
   v
[move_base]
  - GlobalPlanner
  - EBandPlannerROS
   |
   v
/hf_platform/nav_vel
   |
   v
[twist_mux]
   | \
   |  \__ /hf_platform/joy_vel
   |
   \____ /hf_platform/crg_vel
   |
   v
/hf_platform/twist_mux/cmd_vel
   |
   v
[communication_rs485/rs485_control]
   |
   v
[底盘]
```

同时并行存在：

```text
[map_server + cartographer + lidar_odometry]
   -> /amcl_pose_tf

[sick + laser merger + laser filter]
   -> /scan_full_filtered
```

## 10. 当前结构的主要特征

当前旧方案有下面几个明显特征：

1. 底层驱动、建图、定位和 `move_base` 是分层的
2. 底盘最终只关心 `Twist`
3. DMS 不直接关心 `move_base`，而是通过 `/obsNavigation_*` 接口和机器人交互
4. 真正强耦合 `move_base` 的位置在 `task_communication`
5. 充电保持线程和普通导航不是一条执行链
6. 建图链 `hf_karto` 本身不依赖 `move_base`
