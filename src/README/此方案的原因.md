# 此方案的原因

## 1. 文档目的

这份文档只记录两类内容：

1. 第一阶段为什么采用当前方案
2. 为什么不用 MBF、为什么第一阶段不用完整 TEB、以及项目工期为什么这么估

主方案本体仍然以 [NoneMoveBasePlan.md](/data/a/chemist_robot3.0/src/README/NoneMoveBasePlan.md) 为准。

## 2. 第一阶段为什么采用当前方案

第一阶段的目标不是“完整替掉旧链”，而是：

1. **3 天内先让机器人动起来**
2. 跑通一条独立于旧 `move_base` 的最小闭环
3. 先把后续要保留的接口边界定住

所以第一阶段方案收敛为：

1. 全局：**二维占据栅格 A\***
2. 局部：**轻量 Holonomic Path Tracker**
3. 启动方式：**`parallel_debug.launch`**
4. 输出链：**`/hf_platform/nav_vel -> twist_mux -> rs485_control`**

### 2.1 为什么第一阶段全局先用 A\*

第一阶段全局先用 A\*，原因很直接：

1. 你当前已经稳定提供 `/map`
2. 第一阶段只需要在二维占据栅格上生成一条可跟踪路径
3. 全向麦轮的运动学约束重点在 local，不在 global
4. A\* 自包含、依赖少、实现和调试都快
5. 3 天目标下，不值得先引入复杂外部规划框架

对第一阶段来说，A\* 已经足够支撑：

1. 从当前位姿到目标位姿生成栅格路径
2. 做基础去锯齿和重采样
3. 给 local tracker 提供稳定的参考路径

### 2.2 为什么第一阶段局部先用轻量 Holonomic Path Tracker

第一阶段 local 先用轻量 Holonomic Path Tracker，核心原因是它最符合当前约束：

1. 你的底盘本来就支持 `linear.x + linear.y + angular.z`
2. tracker 可以直接输出这三个自由度，不需要再绕一层差速近似
3. 它是 standalone 控制器，容易放进新的运行时边界
4. 它的代码量和调试面都明显小于完整 TEB
5. 它足够支撑第一阶段的“先能走、先到站、先闭环”

第一阶段真正要验证的是：

1. 新 global/local 链路能不能成立
2. `/hf_platform/nav_vel` 能不能稳定接到底盘
3. 到站、停障、速度限制这些最基本语义能不能跑通

tracker 更适合先完成这件事。

### 2.3 为什么第一阶段不用完整 TEB

不是 TEB 不好，而是它不适合作为“3 天能动版”的第一落点。

原因有 5 个：

1. TEB 更适合做第二阶段的增强型 local controller，不适合做第一阶段的最小闭环
2. 你当前目标是先脱离 `move_base` 体系，而不是先把最强 local 算法一次接完
3. 完整 TEB 真正落地时，麻烦不在算法名，而在 standalone 化、输入适配、参数整定和真机验证
4. 第一阶段时间最紧，任何超出“路径跟踪 + 基本避障 + 速度输出”的复杂度都会直接拉长周期
5. 第一阶段先用 tracker，把接口留好，第二阶段再升级到更强 local，返工最小

第一阶段如果强行上完整 TEB，实际要一起解决：

1. 路径输入和重规划节奏
2. 障碍输入和障碍代价接线
3. footprint、速度、加速度、终点姿态约束
4. standalone 运行时封装
5. 大量真机参数整定

这些事不是 3 天问题，而是第二阶段的主要工作量之一。

另外还有一个很实际的原因：

1. 阶段 1 先用 tracker，就能把“终点收敛”问题拆成确定的小问题
2. 也就是：
   - 路径重采样
   - near-goal
   - 最终 yaw 对齐
   - success latch
3. 这几项先跑通后，第二阶段再替换 local 内核，接口和判停语义都不用重定

一句话收敛：

**第一阶段先用轻量 Holonomic Path Tracker，不是因为它是最终最优，而是因为它是“3 天内把新链跑起来”的最短路径。**

## 3. 为什么不用 MBF

这里直接说结论：

**不是 MBF 不好，而是它解决的问题和当前最终目标不一致。**

MBF 适合解决的问题是：

1. 不想继续跑原 `move_base`
2. 但还想保留 `move_base` 兼容接口
3. 还想继续沿用 `nav_core` / plugin 风格的 planner / controller 编排

而你现在已经确认的目标是：

1. **彻底脱离 `move_base` 体系**
2. 外部只保留 DMS 业务接口
3. 内部改成自己的任务执行结构
4. 后面还要承接 `charge / uncharge / back-off / rotate / charge-approach`

所以不用 MBF 的原因有 5 个：

### 3.1 它本质上仍然属于 move_base 兼容体系

MBF 的价值就是兼容旧导航生态。

这和你当前要的“彻底脱离”目标相反。

如果用了 MBF：

1. 你只是把原 `move_base` 节点换成了另一个兼容壳
2. 但内部思维模型还是：
   - action goal
   - planner plugin
   - controller plugin
   - recovery plugin

这不是你现在要的最终边界。

### 3.2 它不能消掉旧 task_communication 的结构性耦合

当前真正耦合 `move_base` 的，不只是 planner/controller。

关键是旧 [task_communication](/data/a/chemist_robot3.0/src/platform/hf_nav/task_communication) 自己就是基于 `MoveBaseAction` 写的。

如果用了 MBF：

1. 你可以短期少改一些
2. 但最终仍然要重写 `task_communication`

也就是说：

1. MBF 不是最终方案
2. 只是一个过渡方案
3. 会把最终必须做的接口重构推迟，而不是消除

### 3.3 它对你的特殊业务流没有本质帮助

你项目真正复杂的地方不是“从 A 点到 B 点发一条 path”。

真正复杂的是：

1. `move`
2. `charge`
3. `uncharge`
4. `back-off`
5. `rotate-in-place`
6. `charge-approach`
7. DMS JSON 状态回传
8. `/hf_platform/crg_vel` 保持线程

这些都属于业务执行编排，不是 MBF 的强项。

所以即便上 MBF：

1. 你仍然要自己写业务执行器
2. 仍然要自己处理 docking/charge 流程
3. 仍然要自己维护外部 JSON 状态接口

### 3.4 它会把运行时边界搞得不彻底

如果现在选 MBF：

1. 外部是你的 DMS 业务接口
2. 中间是 MBF / plugin 生态
3. 下面才是你自己的算法和底盘链

这会让最终系统的运行时边界分成三层。

而方向 B 的价值就在于：

1. 外部保留旧业务接口
2. 内部从 adapter 开始就是你自己的体系
3. global/local 只是被你调用的算法模块

边界更干净，后面演进成本更低。

### 3.5 用 MBF 会更快上线，但会拖慢最终落地

这点最现实。

如果你的目标是：

1. 先快速替掉原 `move_base`
2. 后面再慢慢重构

那 MBF 是对的。

但你已经明确不要这个路径，而是要一次性转到彻底脱离结构。

那就不该再引入一个过渡壳。

一句话收敛：

**不用 MBF，不是因为 MBF 不够强，而是因为它属于“过渡期最优”，不是“最终架构最优”。**

## 4. 项目落地时间估计

这里按 **1 名熟悉 ROS1/C++ 的工程师全职推进** 估算，并且按真实机器人项目来算，不按“算法 demo”来算。

### 4.1 保守估计

如果按两阶段看：

1. 第一阶段“先动起来”：**3 天**
2. 第二阶段“替掉旧链做日常导航”：**额外 5 到 8 周**

从开始建包到可以替掉旧链做日常导航，建议按：

1. **6 到 10 周**

如果中间还包含：

1. 现场机器人联调时间受限
2. 充电站靠泊反复调试
3. 动态障碍场景参数整定
4. 多轮回归验证

那建议按：

1. **8 到 12 周**

### 4.2 分阶段工期

#### 阶段 1：3 天能动版

内容：

1. 建最小包骨架
2. 跑通 `NavigateTask`
3. 跑通 `A* -> Path`
4. 跑通轻量 local tracker 输出 `/hf_platform/nav_vel`

时间：

1. **3 天**

#### 阶段 2：局部控制升级与稳定化

内容：

1. 把更强的 local controller 运行逻辑移入 `none_move_base_local`
2. 接上 `/scan_full_filtered`、`/odom`、`/amcl_pose_tf`
3. 处理速度限制、加速度限制、终点姿态

时间：

1. **2 到 3 周**

这是第一大头。

#### 阶段 3：业务接口与状态机迁移

内容：

1. 替掉旧 `task_communication`
2. 兼容 `/obsNavigation_in`
3. 兼容 `/obsNavigation_out`
4. 兼容 `/inquiryNavigation_out`
5. 保持旧状态字段和业务语义

时间：

1. **1 到 1.5 周**

#### 阶段 4：特殊行为迁移

内容：

1. `back-off`
2. `rotate-in-place`
3. `charge`
4. `uncharge`
5. `charge-approach`
6. `/hf_platform/crg_vel` 保持链接入

时间：

1. **1.5 到 2.5 周**

这是第二大头。

#### 阶段 5：联调、整定、回归

内容：

1. 静态场景导航
2. 人员动态遮挡
3. 窄通道
4. 工位前对位
5. 充电站流程
6. 长时间运行稳定性

时间：

1. **1.5 到 2.5 周**

### 4.3 为什么需要这么久

时间长，不是因为：

1. A\* 难写
2. topic 难接

真正耗时的是下面这些：

#### 4.3.1 不是在写 demo，而是在替生产链

你不是做一个 RViz 演示。

你是在替：

1. 任务接口
2. 日常导航
3. 充电流程
4. 底盘真实运动

生产链的要求是：

1. 能跑
2. 稳定
3. 可恢复
4. 能回归

#### 4.3.2 第二阶段 local 升级是硬成本

3 天内先动起来，可以用轻量 tracker。

真正耗时的是第二阶段把它升级成更稳定的 local planner。

要调的至少包括：

1. `vx / vy / wz` 限幅
2. 加速度约束
3. 障碍物代价权重
4. 终点姿态权重
5. 到站阈值
6. 狭窄区域行为

这些都必须上真机验证。

#### 4.3.3 charge 相关不是普通导航

普通到站和充电前靠泊不是同一类问题。

充电链会引入：

1. 特殊停止条件
2. 特殊对位行为
3. 特殊失败恢复
4. 与 `/hf_platform/crg_vel` 的切换时序

这部分不能靠“复用 planner”自动解决。

#### 4.3.4 旧代码不能机械搬运

你虽然保留旧业务语义，但旧实现里很多动作原语就是 `MoveBaseGoal`。

所以不是 copy 一份函数就结束，而是：

1. 重新定义动作原语
2. 重新拼装状态机
3. 重新定义成功/失败条件

#### 4.3.5 真正难的是边界收敛

这个项目的难点不在单个算法，而在：

1. 哪些接口保留
2. 哪些接口废弃
3. 哪些逻辑迁移
4. 哪些逻辑继续复用

边界没收敛好，后面返工会非常重。

### 4.4 工期判断

如果你的目标是：

1. 先跑通一版可演示的 debug 链

那 **3 天** 可以看到第一版“能动起来”的结果。

如果你的目标是：

1. 真的替掉旧链做日常任务

那我建议从一开始就按：

1. **6 到 10 周**

来做排期，而不要按 2 周 demo 去估项目。
